## Challenge: AAMM v7 Tri-Agent Implementation Plan

### Will Break
- [B1] Task 9 Step 1.1 (Gemini file request invocation) — trigger: The bash script cats `gemini-file-request.md` and applies sed for some placeholders (e.g., {OWNER}, {REPO}), but does not handle injection of dynamic content like {MANIFEST_JSON}, {KB_ECOSYSTEM_CONTENT}, etc., as noted in the comment "# Inject manifest + KB content". At runtime, the prompt will contain unresolved placeholders, causing Gemini to receive invalid input and likely fail to output a valid JSON array. — blast radius: Phase 1 fails for Gemini, blocking all downstream consensus phases; scans cannot proceed without file requests, leading to incomplete or aborted scans. — fix: Add explicit steps to build the full prompt using a tool like `jq` or multi-line sed/cat/echo to inject JSON/YAML content, e.g., `MANIFEST_JSON=$(cat $SCAN_DIR/manifest.json)` then `PROMPT=$(echo "$PROMPT" | sed "s|{MANIFEST_JSON}|$MANIFEST_JSON|g")`, and write the assembled prompt before invocation.
- [B2] Task 8 Step 0.4 (Detect repo type) — trigger: The plan states "Claude (orchestrator) reads the cloned repo and determines repo_type" but provides no mechanism for Claude to access or analyze the clone (e.g., no bash commands to list files, no subagent dispatch with file reading tools). At runtime, the orchestrator (Claude) cannot execute this without assuming undefined tools, leading to a stuck state or manual intervention. — blast radius: Phase 0 halts, preventing manifest generation and all subsequent phases; no scans can complete. — fix: Add explicit bash or subagent steps, e.g., `find $SCAN_DIR/clone -type f > $SCAN_DIR/file_list.txt` then dispatch a Claude subagent with input: "Read file_list.txt and apply detection rules to output repo_type.txt".
- [B3] Task 10 Step 2.2 (Gemini analysis invocation) — trigger: The `gemini` command uses `-o text` but the output is expected to be JSON; if Gemini outputs anything non-JSON (e.g., due to hallucination or prompt misinterpretation), the subsequent "Parse JSON from output" step will fail without specified error handling. — blast radius: Phase 2 fails for Gemini, triggering PARTIAL mode unnecessarily; affects consensus quality and report accuracy for all scans. — fix: Add JSON validation post-invocation, e.g., `cat $SCAN_DIR/phase-1/gemini-analysis-raw.md | jq . > $SCAN_DIR/phase-1/opportunity-map-gemini.json || echo "JSON parse failed" && retry with a format-correction prompt`.
- [B4] Task 11 Step 3.2 (Consensus loop for Claude) — trigger: The JSON example for Claude's scoring output includes "item_id" but the plan doesn't specify how Claude (as subagent) is dispatched or prompted for this; it's just a JSON snippet without invocation code, unlike Gemini/Grok. At runtime, no Claude scores are generated, breaking the tri-agent loop. — blast radius: Consensus loop incomplete, leading to all items stuck as pending; no approved opportunities, resulting in empty reports. — fix: Add symmetric dispatch for Claude, e.g., "Dispatch Claude subagent with prompts/gemini-consensus-round.md adapted for Claude, input: FINDINGS_JSON + PREVIOUS_ROUNDS, output: phase-1/claude-round-$N-scores.json".
- [B5] Task 17 Step 3 (Test clone on small repo) — trigger: The clone command uses `$GITHUB_TOKEN` in the URL but git clone with HTTPS+token can fail if the token lacks 'repo' scope or if the repo is private and token is expired; the script echoes "Clone exit: $?" but doesn't check it, so a failed clone (exit 128) will proceed to `ls` which errors. — blast radius: Smoke test falsely passes despite clone failure, leading to uncaught bugs in production clones across all scans. — fix: Add explicit check: `if [ $? -ne 0 ]; then echo "Clone failed"; exit 1; fi` after the git clone.
- [B6] Task 8 Step 0.3 (Clone repository) — trigger: For learning scans, `--shallow-since="12 months ago"` fetches only recent history, but if KB patterns require older commits (e.g., long-term adoption signals), analysis will miss evidence; no fallback to deeper clone if needed. — blast radius: Learning scans produce incomplete KB proposals, degrading model evolution over time. — fix: Add a conditional deeper clone, e.g., if initial clone has <500 commits, re-clone with `--shallow-since="24 months ago"`.

### Will Strain
- [S1] Task 8 Step 0.3 (Clone repository) + Task 18 squash — at scale: Cloning 100+ repos at 3 AM with `--depth=100` will hit GitHub rate limits (5000 API calls/hour, but clone uses git protocol which counts toward it indirectly via auth); disk usage balloons to 100GB+ for large repos, filling /tmp on a standard machine. — mitigation: Implement clone caching (e.g., check if /tmp/clone exists and git pull instead) and add rate-limit backoff (e.g., sleep 60s between clones); use a dedicated volume for $SCAN_DIR.
- [S2] Task 9 Step 1.3 + Task 10 Step 2.3 (Grok batching) — at scale: For repos with 10,000+ files, Grok's max 5 batches × 50 files = 250 files limit causes frequent PARTIAL flags; each batch invocation adds 5-10s latency × 5 = 50s overhead per scan, totaling 80+ minutes for 100 scans. — mitigation: Increase batch size to 100 if xAI API supports larger contexts, or parallelize batches with multiple API keys; add a pre-scan file count check to warn/skip oversized repos.
- [S3] Task 11 Step 3.2 + Task 12 Step 4.2 (Consensus loops) — at scale: Max 5 rounds × 3 agents × 20 pending items = 300 API calls per scan; for 100 scans, that's 30,000 calls, exceeding xAI/Gemini daily limits (assume 10k/day) and costing $50+ in API fees. — mitigation: Optimize by batching items into fewer calls (e.g., one prompt per round scores all items) and add early exit if no score changes after round 3; monitor API usage with a global counter.

### Solid
- Task 1-5 (Infrastructure & Scaffolding): These create isolated files (ADR, personas, schema, config) with atomic commits; no interdependencies, and bash commands are simple and verifiable, ensuring a stable base before prompts and skills.
- Task 6-7 (Prompt files): Directory creation and wildcard git add work reliably; prompts are self-contained and based on v6 templates, reducing risk of syntax errors.
- Partial handling in Tasks 10-12: Symmetric retry (5x) + continue with WARNING is robust for flaky APIs at 3 AM, preventing total scan failure while logging issues.

### Design Questions
- How does the orchestrator (Claude) handle monorepo subproject selection in Task 8 Step 0.5 if churn ties or no subprojects meet the 5-commit threshold—does it default to root, or abort?
- In learning mode (Task 13), what prevents infinite file requests since "No file quota" is specified—how is request looping bounded to avoid API abuse?
- For consensus:false items in Phase 3/4, are they included in reports (e.g., for CoE review), or dropped entirely? The plan mentions preserving positions but not output handling.
- What happens if all agents fail health checks in Phase 0—does the scan retry automatically, or require manual restart? No recovery loop is defined.
- In Task 14 (select-reference-repos), how does Gemini/Grok "research via web + GitHub API" without specified tools or auth—assumes agents have built-in capabilities, but plan lacks implementation details.
- For squash in Task 18, how to handle conflicts if main changes during the 18-task implementation—rebase assumes no external merges, but no conflict resolution steps are provided.
