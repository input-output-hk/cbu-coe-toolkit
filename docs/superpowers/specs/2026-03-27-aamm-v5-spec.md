# AAMM v5 — Specification

> **Status:** Revised after adversarial review rounds 1 and 2. Pending final review.
> **Date:** 2026-03-27
> **Author:** Dorin Solomon (CoE lead) + Claude (AI agent)
> **ADR:** ADR-018 (supersedes ADR-017)
> **Context:** After 9 sessions of v4 pipeline refinement, fundamental redesign. Single AI agent replaces bash pipeline + adds Knowledge Base for cross-repo learning.

---

## 1. What AAMM Is

**AAMM is an AI-powered consultation per repo.** It tells teams where they are on AI readiness and adoption, and what to do next with the highest ROI.

AAMM produces **coarse-grained indicators** (4 levels per pillar/zone) rather than numeric scores. The goal is directional guidance and actionable recommendations, not measurement precision. The indicators, skill tree, and quadrant ARE assessment tools — but they serve recommendations, not rankings.

### Four Problems AAMM Solves

| # | Problem | How AAMM Solves It |
|---|---------|-------------------|
| 1 | **Awareness gap** — teams don't know what AI can do across the full SDLC | Agent knows the full landscape, compares with what exists, shows the gap |
| 2 | **Where to start** — teams don't know what to do first for their specific context | 3-5 recommendations, ROI-prioritized, ecosystem-specific |
| 3 | **No feedback loop** — learnings stay with individual teams, no cross-pollination | Knowledge Base captures what works across the portfolio and beyond |
| 4 | **Risk without guardrails** — AI on high-assurance code without boundaries is dangerous | Agent identifies missing trust boundaries, critical paths without protection |

### Three Audiences

| Who | What they see | Time investment |
|-----|-------------|-----------------|
| **Team / tech lead** | Top 3-5 recommendations + risk flags | 5 minutes |
| **CoE lead** | Portfolio view: where is everyone, who needs help | 15 minutes |
| **Leadership** | Evolution: delta between scans, portfolio trend | 2 minutes |

### What AAMM Does NOT Do

- Does not measure code quality (that's CMM)
- Does not measure business impact (that's Engineering Vitals)
- Does not judge teams — informs and recommends
- Does not produce numeric scores — produces leveled indicators with evidence and confidence

## 2. Architecture

### Single Active Component

```
┌──────────────────────────────────────────────┐
│                  AAMM v5                      │
│                                               │
│  ┌───────────┐         ┌──────────────────┐  │
│  │ Knowledge │◀───────▶│    AI Agent      │  │
│  │ Base      │ reads   │                  │  │
│  │           │+updates │ - explores repo  │  │
│  └───────────┘         │ - rubric assess  │  │
│                        │ - depth assess   │  │
│                        │ - recommends     │  │
│                        │ - identifies risk│  │
│                        └────────┬─────────┘  │
│                                 │             │
│                           ┌─────▼─────┐      │
│                           │  Report   │      │
│                           └───────────┘      │
└──────────────────────────────────────────────┘
```

One active component: **the AI agent**. Knowledge Base is memory. Report is output.

The agent follows a **rubric + depth methodology** (see Section 3a) — structured evaluation criteria anchor each level, then qualitative exploration adds findings and nuance. Everything is done by the agent; the rubric is part of the prompt, not a separate system.

### Why Single Agent, Not Dual Architecture

ADR-017 proposed dual architecture: deterministic pipeline for scores + AI agent for findings. ADR-018 supersedes:

1. **The pipeline was a workaround.** All v4 complexity (50-call budget, sampling, grep patterns, signal weights) existed because bash can count but cannot understand.
2. **Reproducibility via rubric, not pipeline.** The agent follows defined rubric criteria per pillar/zone. Objective criteria produce consistent results; semi-objective criteria are reproducible but carry confidence qualifiers.
3. **Maintaining two systems is waste.** Pipeline + agent means two things to maintain, sync, and debug.
4. **ADR-017's insight is preserved.** "Use each mechanism for what it's best at" — the rubric IS the structured mechanism, the depth IS the AI mechanism. Same principle, unified system.

### Two Execution Paths

| Path | Cost | Use case |
|------|------|----------|
| **Web-based** (Claude Max/Enterprise) | Flat rate subscription | Individual repo scans, on-demand, interactive |
| **API-based** (Claude API) | Per-token cost | Automated portfolio scans, CI integration |

Both paths use the same agent prompt, KB, and report format. Output is identical.

**API cost estimate:** ~50K-100K input tokens + ~10K-20K output tokens per repo. Full portfolio (29 repos): ~1.5M-3M input + ~300K-600K output tokens. With adversarial review, approximately double. Monthly cost at current API pricing: ~$60-120. Web-based path: included in subscription.

### What We Keep from v4

- Language awareness per ecosystem (Haskell, Rust, TypeScript, Python, Lean)
- The quadrant concept (simplified)
- Adversarial review principle (ADR-012) — now applied to assessment AND recommendations
- ADR process for decisions
- Per-repository assessment (org-level is derived, never independently assessed)

### What We Eliminate from v4

- `collect-all.sh`, `score-readiness.sh`, `score-adoption.sh`, `review-scores.sh`, `generate-report.sh`
- Weighted composite scores (0-100)
- Signal weights and formulas
- Boundary logic debates (≥50 vs >50)
- Weight redistribution when signals are added/removed
- Fixed API call budgets
- The global `HAS_AI_ACTIVITY` boolean (v4's architectural flaw — replaced by per-zone independent rubrics)

## 3. What the Agent Evaluates

Two axes: **Readiness** (can AI be productive here?) and **Adoption** (where is AI present in the SDLC?).

### 3a. Assessment Methodology: Rubric + Depth

Every pillar and zone is evaluated in two phases by the same agent:

```
Phase 1 — RUBRIC (anchoring):
  Agent evaluates a defined checklist of concrete criteria.
  Each criterion is YES/NO or count-based.
  Rubric score → determines the status level.

Phase 2 — DEPTH (qualitative exploration):
  Agent reads actual file content, evaluates quality, follows leads.
  Depth does NOT change the status level.
  Depth produces findings with file paths + content excerpts as evidence.
  Depth findings may flag issues for CoE lead override consideration.

Output per indicator:
  Status: 🟢 Practiced (rubric: 4/5)
  Rubric: S1 ✓, S2 ✓, S3 ✓, S4 ✓, S5 ✗
  Depth findings:
    "CI pipeline is comprehensive — build, test, lint, format all
     enforced on PR. But single-package structure limits AI's ability
     to work on isolated modules. [.github/workflows/ci.yml:L12-L45]"
  Confidence: HIGH (4/5 objective criteria met)
```

**Key rule: rubric determines level, depth adds findings.** Depth never changes the level. If depth reveals something the rubric misses, the finding is reported and the CoE lead can add an override for the next scan.

**Why this works:**
- **Reproducibility:** Rubric criteria are structured. Objective criteria produce identical results across runs. Semi-objective criteria are reproducible within a confidence band.
- **Depth:** Agent adds qualitative insight bash couldn't provide.
- **Auditability:** Every level assignment traces directly to rubric criteria.
- **Evolution tracking:** Delta on rubric criteria is comparable across scans.

### 3b. Confidence Model

Every assessment includes a confidence level based on evidence type:

| Evidence type | Examples | Confidence ceiling |
|--------------|---------|-------------------|
| **Objective** | File exists, config parsed, count verified | HIGH |
| **Semi-objective** | File content matches expected structure, pattern detected | MEDIUM |
| **Subjective** | Quality judgment, usefulness evaluation | LOW |

The agent cannot self-assign HIGH confidence on subjective evaluations.

**Grounding requirement:** Every finding must include a file path and content excerpt. The adversarial agent spot-checks 3-5 claims by re-reading cited files. Ungrounded findings are marked LOW confidence.

### 3c. Status — Skill Tree

Two scales: one for readiness (5-criteria rubrics), one for adoption (3-criteria rubrics).

**Readiness status (5-criteria pillars):**

| | Status | Rubric score |
|---|---|---|
| ⬜ | **Undiscovered** | 0-1 of 5 criteria met |
| 🟡 | **Exploring** | 2-3 of 5 criteria met |
| 🟢 | **Practiced** | 4-5 of 5 criteria met |
| 💎 | **Mastered** | 5/5 criteria met AND depth confirms excellence AND CoE lead confirms |

**Adoption status (3-criteria zones):**

| | Status | Rubric score |
|---|---|---|
| ⬜ | **Undiscovered** | 0 of 3 criteria met |
| 🟡 | **Exploring** | 1 of 3 criteria met |
| 🟢 | **Practiced** | 2-3 of 3 criteria met |
| 💎 | **Mastered** | 3/3 criteria met AND depth confirms excellence AND CoE lead confirms |

Each indicator has: **status + rubric score + confidence + evidence**.

**💎 Mastered** is special:
- Agent nominates with evidence
- CoE lead confirms (human gate — evaluates whether the practice is genuinely exemplary and transferable)
- Mastered repos feed the Knowledge Base
- If a Mastered repo regresses (detected at next scan), the nomination is revoked and KB entry flagged for review

### 3d. Readiness — 5 Pillars

| Pillar | Question | What the agent explores |
|--------|----------|------------------------|
| **Structure** | "Is the code organized for collaboration?" | File organization, module boundaries, build system, dependency pinning, linting/formatting |
| **Clarity** | "Is it clear how the code works?" | Type safety, doc comments quality, schemas/contracts |
| **Purpose** | "Is it clear what we're building, why, and for whom?" | PRD/FRD in repo, ADRs, ARCHITECTURE.md, DoD, acceptance criteria, cross-repo context |
| **Workflow** | "Is it clear how the team works?" | PR process, branching, CONTRIBUTING.md, CODEOWNERS, trust boundaries, release process |
| **Safety Net** | "Can changes be verified?" | Test diversity + quality, CI test enforcement, CI quality gates, coverage |

**Why 5, not 3:** v4 had 3 pillars because bash couldn't evaluate Purpose or Workflow. An AI agent can — it reads PRDs, ADRs, CONTRIBUTING.md and evaluates whether they contain useful information.

**Overlap resolution:** Where an exploration item could belong to multiple pillars, the primary owner is:
- CI pipeline configuration → **Structure** (it's infrastructure)
- CI enforcement of quality gates (tests block merge) → **Safety Net** (it's verification)
- ARCHITECTURE.md → **Purpose** (it's about intent and decisions)
- Schemas/contracts → **Clarity** (it's about understanding interfaces)
- .aiignore / trust boundaries → **Workflow** (it's about how AI should work here)

**Cross-pillar comparison caveat:** Pillar difficulty varies. Structure criteria (linter, formatter, CI) are common in mature repos. Purpose criteria (ADRs, PRD, DoD) are rarer across the industry. A repo with Structure: Practiced and Purpose: Exploring is normal, not necessarily underinvesting in Purpose. The report contextualizes this.

#### Structure — Rubric (5 criteria)

| # | Criterion | Check | Type |
|---|-----------|-------|------|
| S1 | Module boundaries exist | cabal.project / Cargo workspace / pnpm workspace / multiple package manifests. **Single-package repos:** if the repo is a deliberately scoped library (not a monolith), this criterion passes with a depth note. | Objective |
| S2 | Build system reproducible | Lockfile present + runtime version pin (.nvmrc, rust-toolchain.toml, .python-version, cabal.project freeze) | Objective |
| S3 | CI pipeline active | CI workflow files present + last run within 30 days | Objective |
| S4 | Linter configured | Ecosystem-appropriate linter config (hlint, eslint, clippy, ruff, etc.) | Objective |
| S5 | Formatter configured | Ecosystem-appropriate formatter config (fourmolu, prettier, rustfmt, black, etc.) | Objective |

Depth: Agent evaluates file organization quality, directory consistency, whether linter/formatter are CI-enforced, build reproducibility quality.

#### Clarity — Rubric (5 criteria)

| # | Criterion | Check | Type |
|---|-----------|-------|------|
| C1 | Strong type system or strict mode | Haskell/Rust/Lean = auto-pass; TypeScript strict:true; Python type hints + mypy | Objective |
| C2 | Doc comments present on public API | Sample 5-10 source files, >50% of public items have doc comments | Semi-objective |
| C3 | README explains what the project does | README.md exists with >100 words of actual content (not just badges) | Semi-objective |
| C4 | Schema/contract definitions exist | .proto, .graphql, .cddl, zod schemas, contract/ packages, servant types | Objective |
| C5 | Linter enforces code style rules | Linter config includes style/naming rules (ESLint naming-convention, HLint, clippy::style) | Objective |

Depth: Agent reads doc comments for quality (not just presence), evaluates naming consistency, assesses whether schemas cover module boundaries.

#### Purpose — Rubric (5 criteria)

| # | Criterion | Check | Type |
|---|-----------|-------|------|
| P1 | Architecture documented | ARCHITECTURE.md or architecture section in README with real content (>100 words) | Semi-objective |
| P2 | ADRs present (≥3) | docs/decisions/ or adr/ directory with ≥3 decision records | Objective |
| P3 | Product context in repo | PRD, FRD, specs, user stories, or requirements docs accessible in repo tree | Objective |
| P4 | Definition of Done visible | DoD with user/acceptance perspective in templates or docs | Semi-objective |
| P5 | Cross-repo context documented | README or docs explain ecosystem position, dependencies, interface contracts | Semi-objective |

Depth: Agent reads ADRs for quality (are decisions explained with rationale?), evaluates whether product context is sufficient for AI to understand scope.

#### Workflow — Rubric (5 criteria)

| # | Criterion | Check | Type |
|---|-----------|-------|------|
| W1 | CONTRIBUTING.md exists with substance | File present, >50 words, explains how to contribute | Semi-objective |
| W2 | PR template exists | .github/PULL_REQUEST_TEMPLATE.md or equivalent | Objective |
| W3 | Issue templates exist | .github/ISSUE_TEMPLATE/ with ≥1 template | Objective |
| W4 | CODEOWNERS defined | CODEOWNERS in root, .github/, or docs/ | Objective |
| W5 | Trust boundaries documented | .aiignore, or CLAUDE.md/AI config with explicit critical path exclusions | Objective |

Depth: Agent evaluates PR review patterns (from recent PRs), branching conventions, merge strategy, release process, commit conventions.

#### Safety Net — Rubric (5 criteria)

| # | Criterion | Check | Type |
|---|-----------|-------|------|
| SN1 | Tests exist | Test directories or test files present in repo tree | Objective |
| SN2 | Multiple test categories | ≥2 distinct test types visible in directory structure or CI config (unit, integration, e2e, property) | Semi-objective |
| SN3 | Tests run in CI | CI workflow contains test execution steps | Objective |
| SN4 | Tests block merge | Recent merged PRs show CI status checks (from PR data) | Semi-objective |
| SN5 | Coverage tooling present | Coverage tool in CI + threshold configuration | Objective |

Depth: Agent reads sample tests for quality (happy path only? edge cases? property tests?), evaluates CI pipeline comprehensiveness.

### 3e. Adoption — 5 Zones

| Zone | Question | What the agent looks for |
|------|----------|--------------------------|
| **Code** | "Does AI participate in code?" | AI config, co-authored commits, AI PRs, AI code review |
| **Testing** | "Does AI help with testing?" | AI-attributed test changes, AI test tooling |
| **Security** | "Are there guardrails for AI?" | .aiignore, trust boundaries, AI security review |
| **Product & Delivery** | "Does AI help with product and delivery?" | AI changelogs, AI PR summaries, AI issue management |
| **Governance & Architecture** | "Is AI managed maturely and does it challenge decisions?" | Multi-tool config, attribution, AI as reviewer, orchestration |

#### Adoption Detection: What's Detectable vs Inferred

| Zone | Detectable (HIGH confidence possible) | Inferred (LOW confidence ceiling) |
|------|---------------------------------------|----------------------------------|
| **Code** | Co-authored-by tags, bot PR authors, AI config files, AI actions in CI | Whether non-attributed code was AI-assisted |
| **Testing** | AI config mentions testing, bot PRs modifying test files, AI test tools in CI | Whether test files were AI-generated without attribution |
| **Security** | .aiignore exists, trust boundaries in AI config, security-focused AI actions | Whether AI reviews focus on security |
| **Product & Delivery** | AI bot issues, AI changelog generators in CI, AI PR summary tools | Whether PR summaries are AI-written without attribution |
| **Governance & Architecture** | Multi-tool config (≥2 AI tools), AGENTS.md, .claude/commands/, Co-authored-by consistency | Whether AI challenges architecture decisions |

**Rule:** If a zone has only inferred evidence, the maximum status is 🟡 Exploring with LOW confidence.

**Industry limitation acknowledged:** Testing, Security, and Product & Delivery adoption are difficult to detect because most AI tools do not yet provide reliable attribution for these activities. As AI tool attribution improves (e.g., structured co-authored-by per SDLC activity), these zones will become more detectable. This is an industry-wide limitation, not a model flaw. The agent is honest about what it can and cannot observe.

## 4. Recommendations Design

### Constraints

- **Max 3-5 per repo** — only the highest ROI
- **Must survive adversarial review** before appearing in final report
- **Must be measurable** — verifiable at next scan via concrete check where possible

### Recommendation Format

```
RECOMMENDATION #N: [Concise title]

What:       The concrete action
Why:        The problem it solves + concrete impact
Effort:     🟢 Low (days) | 🟡 Medium (weeks) | 🔴 High (months)
Impact:     HIGH / MEDIUM + concrete explanation
Source:     🧠 Model knowledge | 💎 KB pattern (validated)
Measurable: Concrete verification check for next scan
```

### Two Sources of Recommendations

| Source | What it provides | Example |
|--------|-----------------|---------|
| 🧠 **Model knowledge** | General + domain-specific best practices from AI training data | "Property-based testing with QuickCheck is standard for Haskell" |
| 💎 **Knowledge Base** | Validated patterns from IOG repos and external exemplary repos | "cardano-ledger uses QuickCheck generators per module — pattern validated" |

KB-sourced recommendations carry more weight because they are human-validated and context-specific.

### Measurability Requirement

Each recommendation's "Measurable" field must specify a **concrete verification check** where possible:

- ✅ "File `test/Gen*.hs` exists with at least 3 `Arbitrary` instances"
- ✅ "CONTRIBUTING.md exists with >50 words"
- ✅ ".aiignore file lists crypto/ and signing/ paths"
- ⚠ "Test quality improves" → tagged `human-verify`, flagged for manual confirmation at next scan

### Adversarial Review (Mandatory)

The scanner agent produces the assessment + 5-7 draft recommendations. A **separate adversarial agent** challenges both:

```
Scanner Agent                    Adversarial Agent
     │                                │
     │  produces assessment +          │
     │  5-7 draft recommendations     │
     │                                │
     │──────── sends ────────────────▶│
     │                                │
     │          Adversarial agent RE-READS the repo
     │          independently. Has skeptical system prompt.
     │                                │
     │          Challenges ASSESSMENT:
     │            - Spot-check 3-5 rubric criteria
     │              (re-verify S1, P2, AC1, etc.)
     │            - Spot-check 3-5 depth findings
     │              (re-read cited files, verify claims)
     │            - Flag any rubric criterion that seems
     │              incorrectly evaluated
     │                                │
     │          Challenges RECOMMENDATIONS:
     │            - Does it make sense for this ecosystem?
     │            - Is it clear and actionable?
     │            - Is it measurable at next scan?
     │            - Is the team maybe already doing this?
     │            - Is it truly top ROI?
     │            - Does it contradict another finding?
     │                                │
     │◀─────── returns ───────────────│
     │    Rubric corrections (if any) │
     │    Approved recs (3-5)         │
     │    Rejected recs + reasons     │
     │                                │
     ▼
  Final report
  (rubric corrected if needed;
   only approved recommendations;
   rejected saved in detailed log)
```

Rejected recommendations and adversarial dialogue saved in detailed log. Full transparency.

## 5. Report Structure

### Report per Repo

```
1. SUMMARY (30 seconds)
   "lace-platform — TypeScript wallet"
   Readiness: Medium | Adoption: Medium
   Quadrant: Emerging

2. TOP RECOMMENDATIONS (5 minutes)
   3-5 actions, ROI-prioritized
   Each with: what, why, effort, impact, source, measurable

3. RISK FLAGS (2 minutes)
   Critical issues regardless of AI
   (unreviewed PRs, no vulnerability scanning, missing trust boundaries)

4. SKILL TREE (context)
   Readiness × 5 pillars (status + rubric score + confidence)
   Adoption × 5 zones (status + rubric score + confidence)
   Each with evidence summary
   Cross-pillar comparison caveat noted

5. FINDINGS (depth)
   Specific observations with file paths + content excerpts
   What works well + what's missing
   Confidence per finding

6. CROSS-REPO INSIGHTS (from KB)
   "Similar repos solved X with Y"
   Relevant patterns for your ecosystem

7. EVOLUTION (if previous scan exists)
   Rubric criteria changes per pillar/zone
   Stable criteria (objective) shown as comparable delta
   Judgment criteria (semi-objective) shown with confidence qualifier
   Which recommendations were adopted (concrete checks)
   Status level changes

8. EVIDENCE LOG (audit)
   All files read by agent with excerpts
   Rubric evaluation per criterion
   Adversarial review results
   Raw data: PR counts, commit data, CI status
```

      "rubric_score": "1/3",
      "confidence": "HIGH"
    }
  },
  "recommendations": [],
  "risk_flags": [],
  "kb_nominations": [],
  "overrides_applied": {}
}
```

**Versioning:**
- `schema_version`: structure of the JSON. Major bump = breaking change.
- `criteria_version`: rubric criteria definitions. Tracks criterion additions, modifications, removals.
- Evolution reports compare assessments with the same `criteria_version`. If versions differ, the report flags which criteria changed and shows delta only on unchanged criteria.

    source_sample_size: 10
    pr_sample_size: 30
    commit_sample_size: 100
```

**Override semantics:**
- Overrides are human-asserted and carry MEDIUM confidence
- Agent still evaluates the criterion independently and reports both its finding and the override
- If they conflict, the report flags the discrepancy
- Overrides are logged in evidence log with asserting human's identity
- Overrides do not bypass the rubric — they provide an alternative evaluation that the report shows alongside the agent's evaluation

## 10. Scan Flow

```
1. Agent reads KB for this ecosystem
2. Agent reads repo tree, identifies key files
3. Agent evaluates Readiness rubric (5 pillars × 5 criteria)
4. Agent performs Readiness depth (reads files, produces findings)
5. Agent evaluates Adoption rubric (5 zones × 3 criteria)
6. Agent performs Adoption depth (reads AI config content, PR data)
7. Agent identifies risk flags
8. Agent drafts 5-7 recommendations from 2 sources
9. ADVERSARIAL AGENT: re-reads repo, spot-checks rubric + findings,
   challenges recommendations
10. Agent generates report (4 levels of detail)
11. Agent proposes KB updates + Mastered nominations
12. CoE lead reviews KB proposals and Mastered nominations (human)
```

Steps 1-11 are automated. Step 12 is human.

### Exploration Guidelines

To ensure consistent depth across runs:
- **Always read:** repo tree, README, CLAUDE.md (or equivalent), CI workflow files, CONTRIBUTING.md, CODEOWNERS
- **Read if present:** ARCHITECTURE.md, ADRs, .aiignore, PR templates, issue templates, package manifests
- **Sample:** 5-10 source files for doc comment evaluation, 5-10 test files for test quality. For monorepos: sample across packages, not just one.
- **PR data:** Last 20-30 merged PRs for review patterns, AI attribution, bot activity
- **Commit data:** Last 50-100 commits for co-authored-by patterns

### Agent Prompt Structure (Skeleton)

```
SYSTEM PROMPT:
  1. Role: AAMM v5 scanner agent
  2. KB content for this ecosystem (loaded from kb/ files)
  3. Rubric tables (all criteria for all pillars and zones)
  4. Methodology instructions:
     - Phase 1: evaluate each rubric criterion, output YES/NO per criterion
     - Phase 2: read key files, produce depth findings with file:line citations
     - Rubric determines level. Depth does NOT change level.
  5. Confidence rules: objective→HIGH, semi-objective→MEDIUM, subjective→LOW
  6. Grounding rule: every finding must cite file path + content excerpt
  7. Output format: structured JSON (assessment.json schema) + markdown report

USER PROMPT:
  1. Repo identifier (owner/repo)
  2. Repo tree (from GitHub API)
  3. Key file contents (README, CLAUDE.md, CI workflows, etc.)
  4. PR data (recent merged PRs with authors, reviewers, body excerpts)
  5. Commit data (recent commits with co-authored-by trailers)
  6. Previous assessment (if exists, for evolution comparison)
  7. Overrides from config.yaml (if any)
```

## 11. Failure Modes

| Failure | Detection | Handling |
|---------|-----------|---------|
| **Repo inaccessible** | API 403/404 | No assessment. Report contains failure reason only. |
| **API rate limit** | 429 response | Pause + backoff. If persistent, partial report with "incomplete" flag. |
| **Context window exceeded** | Agent runs out of context | Focus on rubric (bounded). Skip depth. Flag "partial depth." |
| **Adversarial rejects ALL recs** | 0 approved | Escalate to CoE lead. Include all rejected with reasons. |
| **KB conflicting patterns** | Agent finds contradictions | Flag conflict. Use most recently validated entry. Propose cleanup. |
| **Unknown language** (Lean, Agda, Nix) | Not in ecosystem profiles | Use cross-cutting criteria only. Flag ecosystem-specific as N/A. |
| **Agent hallucination** | Adversarial spot-check fails | Finding removed. Confidence downgraded. Incident logged. |
| **Zero docs, zero tests, zero CI** | All rubric criteria fail | Valid result: Undiscovered. Recommendations focus on highest-impact first steps. |

## 12. Design Decisions Summary

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | AI agent replaces bash pipeline (ADR-018) | Pipeline was a workaround. Rubric provides structured assessment. |
| 2 | 5 readiness pillars | Purpose and Workflow evaluable by AI agent, not bash. |
| 3 | 5 adoption zones | Consolidated from 8. Code+Quality, Product+Delivery, Governance+Architecture. |
| 4 | Rubric determines level, depth adds findings | Depth never changes level. Reproducibility via rubric. |
| 5 | Confidence ceilings by evidence type | Objective→HIGH, Semi-objective→MEDIUM, Subjective→LOW. |
| 6 | Separate scales for readiness (5-criteria) and adoption (3-criteria) | Different rubric sizes need different thresholds. |
| 7 | Mastered = human-confirmed | Agent nominates, CoE lead confirms. Prevents KB contamination. |
| 8 | KB pre-seeded from 9 sessions | First scan needs cross-repo insights for credibility. |
| 9 | Adversarial reviews assessment AND recommendations | Round 1 proved recommendations-only scope was too narrow. |
| 10 | Two execution paths (web + API) | Web for on-demand, API for automation. Same output. |
| 11 | Criteria versioning for evolution | Prevents false regression from criteria changes. |
| 12 | Grounding requirement | Every finding cites file + excerpt. Adversarial spot-checks. |

## 13. Success Criteria

AAMM v5 is successful when:

1. **Teams act on recommendations** — at least 1 recommendation adopted per repo within 30 days
2. **Assessments are honest** — adversarial review finds <20% issues in final reports
3. **External validation** — 3 randomly selected assessments per quarter manually reviewed by CoE lead; agrees with agent on ≥80% of rubric criteria
4. **KB grows** — at least 5 new validated patterns after first full portfolio scan
5. **Evolution is visible** — rubric delta reports show measurable changes between scans
6. **Leadership gets portfolio view** — quadrant + trends in <2 minutes
7. **No false confidence** — LOW confidence indicators clearly labeled; subjective findings never at HIGH
8. **Recommendations are relevant** — <30% of recommendations rejected by teams as irrelevant
