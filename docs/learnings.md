### 2026-03-20 — AAMM v4 LACE sample report: discovering hidden AI signals

- **PR body text is a critical AI detection surface.** Initial scan checked only PR authors and commit co-author lines — missed "Made with [Cursor](https://cursor.com)" in PR #2172's body. AI attribution increasingly appears in PR descriptions, not just commit metadata. The scan must regex-search recent PR bodies for AI tool signatures (`Made with Cursor`, `Generated by`, `Co-Authored-By` in body text, Copilot Coding Agent boilerplate).
- **Search for bot-authored PRs explicitly, don't rely on the recent-30 list.** The Copilot Coding Agent PR (#2173) was closed (not merged), so it didn't appear in the "most recent merged PRs" sample. Use GitHub Search API (`author:app/copilot-swe-agent`, `author:Copilot`) to catch agent PRs regardless of merge status.
- **Submodule AI config is easy to miss.** LACE's v2 submodule (`input-output-hk/lace-platform`) contains a `.claude/` directory that's invisible from the parent repo's tree API. The scan must: (1) detect submodules via `.gitmodules`, (2) follow submodule SHA to check the referenced repo for AI config files, (3) note when submodule repos are inaccessible (private, token scope).
- **Token scope must include `repo` for private repos.** The `public_repo` scope returns 404 for private repos silently (no "requires auth" hint). Always verify token scopes before scanning orgs with private repos. The LACE scan missed the entire lace-platform AI infrastructure because of this.
- **Multi-tool detection requires multiple search strategies.** LACE has 4+ AI tools (Cursor, Copilot Agent, MCP servers, Claude Code) but no single detection method finds all of them: MCP is in tree, Cursor is in PR bodies, Copilot Agent is a PR author, Claude is in a private submodule. The scan must use a checklist of detection methods, not just one pass.
- **"Active signals without Configured foundation" is the most actionable insight for teams.** When AI is actively producing code but there's no institutional context/governance, the model correctly scores 0 — but the annotation is what matters. Teams with organic AI adoption need the lightest push (create CLAUDE.md) to unlock Configured→Active→value.
- **Dependabot security alerts ≠ dependabot.yml config.** GitHub's automatic security alerts fire for npm CVEs even without a dependabot.yml entry for npm (PR #2148 bumps lodash). But this is reactive-only, not proactive version scanning. The penalty for "no dependency scanning" still applies — document the distinction clearly.

### 2026-03-17 — AAMM v3 design and implementation

- **Two-condition gates are the right default for Stage 1.** Config-only Stage 1 (v1) allowed repos to "game" the model by adding a CLAUDE.md without any actual engineering practice. Requiring both practice active + AI config is more honest and more useful for recommendations.
- **Sub-levels beat continuous scores for adoption.** v2's 0-100 continuous scoring required 35+ formula definitions, was hard for agents to reproduce consistently, and confused stakeholders when "progress score 45" contradicted "Stage 1." Sub-levels (Low/Mid/High) give enough granularity for tracking month-over-month progress without the complexity.
- **Cross-cutting dimensions need explicit labeling.** AI Practices & Governance doesn't fit any single SDLC dimension. When it was distributed across other dimensions, scoring became inconsistent. Making it dimension 7 with explicit "cross-cutting" label resolved this.
- **The quadrant model is the communication tool, not the stages.** "Fertile Ground — Readiness 90, Adoption 5" communicates more in one line than any table of stage scores. The quadrant is what leadership remembers; stages are what teams act on.
- **Next Steps as flywheel.** Always 3 steps, always ordered by impact/effort, always showing projected dimension advancement and composite change. This turns the scan from a measurement event into an action generator.
- **Learning signals belong in sub-levels, not a separate axis.** Adding a "learning sophistication" axis would create a 3D model — too complex. Annotations (static/evolving/self-improving) affecting sub-level assignment give the same insight without the complexity.
