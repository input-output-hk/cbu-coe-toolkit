# ADR-003: Multi-Layer AI Signal Detection Methodology

**Date:** 2026-03-20
**Status:** Accepted
**Context session:** AAMM v4 LACE sample report — discovered that single-pass detection misses significant AI signals

## Context

During the first LACE v4 assessment, the initial scan used the standard detection approach: check tree for AI config files, check recent commits for co-author patterns, check recent merged PRs for bot authors. This missed 3 significant AI signals:

1. **Cursor attribution in PR body** — "Made with [Cursor](https://cursor.com)" in PR #2172 (merged). Not in commit metadata, only in the PR description text.
2. **Copilot Coding Agent PR** — PR #2173 authored by `Copilot` (copilot-swe-agent[bot]). Was closed (not merged), so it didn't appear in the "10 most recent merged PRs" sample.
3. **Submodule AI infrastructure** — `.claude/` directory in `input-output-hk/lace-platform` (referenced as `v2/` submodule). Invisible from parent repo's tree API because submodules are opaque pointers.

These are not edge cases — they represent the 3 main ways AI signals hide:
- In **PR text** (not structured metadata)
- In **non-merged PRs** (outside the standard sample window)
- In **linked repositories** (submodules, monorepo references)

## Decision

Adopt a multi-layer detection methodology for AI signals. The scan must use all 5 detection layers, not just the tree + recent-commits approach.

### Layer 1: Repository Tree (existing)
- Scan for AI config files (CLAUDE.md, .mcp.json, .cursor/rules, etc.)
- Detect `.aiignore`, `.cursorignore`
- Cost: 0 additional API calls (uses existing tree)

### Layer 2: Commit Metadata (existing)
- Scan recent 50 commits for `Co-authored-by` patterns
- Cost: 1 API call (existing)

### Layer 3: PR Author Search (new)
- Use GitHub Search API to find bot-authored PRs regardless of merge status:
  - `author:app/copilot-swe-agent` (Copilot Coding Agent)
  - `author:app/coderabbit-ai` (CodeRabbit)
  - `author:Copilot` (Copilot)
- Cost: 1-3 API calls

### Layer 4: PR Body Text Search (new)
- Scan recent 30 PR bodies for AI tool signatures:
  - `Made with Cursor`, `Made with [Cursor]`
  - `Generated by Claude`, `Generated by Copilot`
  - Copilot Coding Agent boilerplate (`Copilot coding agent`)
  - `@copilot` mentions in PR comments
- Cost: 0 additional calls (PR list already fetched; just parse body text)

### Layer 5: Submodule Traversal (new)
- Parse `.gitmodules` to identify submodules
- For each submodule: fetch the referenced repo's tree at the pinned SHA
- Check submodule trees for AI config files
- Note when submodule repos are inaccessible (private, token scope)
- Cost: 1-2 API calls per submodule (tree + optional config file reads)

### Updated API Budget

| Layer | Calls | Notes |
|-------|-------|-------|
| L1-L2 (existing) | 25-35 | No change |
| L3: PR author search | 1-3 | GitHub Search API |
| L4: PR body parsing | 0 | Reuses existing PR data |
| L5: Submodule traversal | 1-4 | Per submodule: tree + config |
| **Total** | **~30-42** | Still within 50-call budget |

## Consequences

- **Better:** Catches organic AI adoption that doesn't follow institutional patterns (individual developer tool use, experimental agent PRs, cross-repo AI config)
- **Better:** Reduces false negatives — repos with real AI activity won't score 0 without annotation
- **Better:** Annotations become richer — "Active signals emerging without Configured foundation" with specific PR evidence
- **Neutral:** Slightly more API calls (~5-7 more), but still within budget
- **Risk:** PR body text search is heuristic — could false-positive on PRs discussing "cursor position" or "copilot program" (mitigate with specific patterns like `Made with [Cursor]`)
- **Risk:** Submodule repos may be inaccessible — always annotate when detection is incomplete due to access
- **Operational:** Token must have `repo` scope for orgs with private repos. Verify before scanning.

## Implementation Notes

Update `adoption-scoring.md` Section 3 (Data Collection) to include Layers 3-5. Update bot detection patterns to include PR body signatures. Add `.gitmodules` parsing to the scan flow.
